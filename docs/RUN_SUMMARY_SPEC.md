# Run Summary Artifact Specification

## Overview
This document specifies the format, generation, and delivery of the run summary artifact for PMs and customer success stakeholders, as per backlog item "Publish PM- and support-facing run summary artifact".

## Purpose
Provide a concise, non-secret summary of a replayt workflow run that can be reviewed without opening a trace UI. The summary should focus on timestamps, run ID, outcome, and high-level steps—not raw payloads.

## Format
The run summary MUST be in JSON format with the following schema:

```json
{
  "run_id": "string",
  "workflow_id": "string",
  "start_time": "ISO 8601 timestamp (UTC)",
  "end_time": "ISO 8601 timestamp (UTC)",
  "outcome": "string (e.g., 'success', 'failure', 'cancelled')",
  "high_level_steps": ["array of strings"],
  "error_message": "string or null"
}
```

Additional fields MAY be added in the future, but the above are REQUIRED for initial implementation.

## Generation
The summary MUST be generated by the existing `generate_run_summary` function in `src/replayt_opentelemetry_exporter/tracing.py`. This function takes a span, workflow_id, run_id, outcome, high_level_steps, and optional error_message, and returns a `RunSummary` dataclass instance. The span MUST expose `start_time` and `end_time` in nanoseconds (OpenTelemetry Python SDK spans do; the span from `workflow_run_span` is suitable).

The `RunSummary` dataclass MUST be serializable to JSON using standard library methods (e.g., `dataclasses.asdict()` or custom `to_dict()` method).

### Generation without `workflow_run_span`

Integrators who do **not** use **`workflow_run_span`** for the full run **MUST** still pass a **`Span`** that has **ended** so **`start_time`** and **`end_time`** are populated (typically an integrator-owned run span). **`generate_run_summary`** does not read metrics; when **`record_run_outcome`** is used for the same logical run, the boolean **`success`** argument and the **`outcome`** string passed here **MUST** agree per [PUBLIC_API_SPEC.md](PUBLIC_API_SPEC.md) **§5.5.2**.

Runnable documentation: **[docs/examples/record_run_outcome_run_summary.md](examples/record_run_outcome_run_summary.md)** ([PUBLIC_API_SPEC.md](PUBLIC_API_SPEC.md) **§3.5**). Pytest obligations for metrics + summary pairing without **§6** lifecycle traces from **`workflow_run_span`** are in [TESTING_SPEC.md](TESTING_SPEC.md) **§4.7**; Builder checklist cross-reference is [PUBLIC_API_SPEC.md](PUBLIC_API_SPEC.md) **§8** item **19**.

## Redaction Policy
The summary MUST exclude secret-bearing fields. Refer to `docs/SECURITY_REDACTION.md` for the complete redaction policy. Specifically:
- No credentials, API keys, or tokens in any field.
- No PII (personally identifiable information) unless explicitly allowed by policy.
- For LLM-related fields, follow the guidance in `SECURITY_REDACTION.md`.

The implementation MUST reference and enforce this policy when generating the summary.

## Integration Path
The summary MUST be exportable from the same integration path as traces. For example:
- After a workflow run completes, the summary can be generated and written to a file (e.g., `run_summary_<run_id>.json`).
- Alternatively, the summary can be sent to an HTTP endpoint alongside telemetry export.
- The summary generation should be optional and configurable (e.g., via environment variable or function parameter).

## Example Output
An example output MUST be provided in the repository. Options:
- Add `docs/examples/run_summary_example.json` with a realistic example.
- Or add a test fixture in `tests/fixtures/run_summary_example.json`.

The example should demonstrate all required fields and show how redaction is applied (e.g., with placeholder values).

## Acceptance Criteria (Refined)
1. **Format specified**: JSON schema as defined above.
2. **Generation function**: `generate_run_summary` in `tracing.py` produces a serializable `RunSummary`.
3. **Redaction**: Summary excludes secrets; references `SECURITY_REDACTION.md`.
4. **Example output**: Provided in docs or tests as a fixture.
5. **Integration**: Summary can be generated and exported alongside traces without modifying replayt core.
6. **Non-`workflow_run_span` path**: Documentation states that a span with **`start_time`** / **`end_time`** is required, and that **`outcome`** agrees with **`record_run_outcome(success=...)`** when both are used; **[docs/examples/record_run_outcome_run_summary.md](examples/record_run_outcome_run_summary.md)** exists and README links to it per [PUBLIC_API_SPEC.md](PUBLIC_API_SPEC.md) **§3.5**.
7. **Tests for that path**: [TESTING_SPEC.md](TESTING_SPEC.md) **§4.7** is satisfied (success/failure **`record_run_outcome`** metrics without relying on **§6** lifecycle signals from **`workflow_run_span`**, plus a pairing test with **`generate_run_summary`**).

## Notes for Builder
- Do not implement new public APIs; use existing `generate_run_summary` and `RunSummary`.
- Ensure the summary is human-readable and suitable for PMs/support.
- Tests should verify the JSON output matches the schema and redaction rules.
- For metrics without **`workflow_run_span`**, follow [TESTING_SPEC.md](TESTING_SPEC.md) **§4.7** and [PUBLIC_API_SPEC.md](PUBLIC_API_SPEC.md) **§3.5** / **§5.5**; acceptance criteria **6–7** above apply when that path ships.
